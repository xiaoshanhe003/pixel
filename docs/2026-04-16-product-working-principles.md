# Pixel Forge Working Principles

## 核心原则

### 1. 先确认产品模型，再扩展功能

新增功能前，先判断它属于哪一层：
- 文档
- 帧
- 图层
- 画布交互
- 导出

模型不清楚时，不要先堆按钮和面板。

### 2. 功能必须闭环，不能只有入口

每个功能都必须同时具备：
- 明确入口
- 状态变化
- 用户可见结果
- 测试或命令验证

禁止只做一个按钮、一个区域或一段说明文案，而没有真实生效的行为。

### 3. 页面优先级服从工作流

界面组织顺序应始终围绕主流程：
- 先画
- 再调
- 再管理图层 / 帧
- 最后导出

说明型内容必须让位于：
- 画布
- 工具
- 图层
- 时间线
- 属性检查

### 4. 画布必须是视觉中心

后续页面调整都应遵守：
- 中间最大区域给画布
- 高频操作靠近画布
- 图层和时间线持续可见
- 不让标题、介绍、营销式文案占主视觉

### 4.1 按钮系统必须先定义，再落到页面

后续所有新按钮和按钮改造，先遵守统一分级，不允许按功能点临时重画。

当前按钮系统固定为四类：

- 一级按钮：`Primary`
- 二级按钮：`Secondary`
- 三级按钮：`Tertiary`
- 色板按钮：`Swatch`

禁止新增下面这些情况：

- 同一个页面里同时出现两种不同圆角体系
- 同一级按钮在不同区域里突然换字号、描边或内边距
- 因为某个局部需求，直接在业务组件里重写按钮底色、hover、active
- 用“是否选中”临时创造新的按钮层级

如果某个场景确实不适配现有层级，应该先修改按钮规范，再修改页面实现。

### 4.2 三个主按钮层级定义

三个主按钮共享这些基础规则：

- 统一圆角：胶囊圆角
- 统一排布：`inline-flex`，水平居中，图标和文字同一行
- 统一字号：正文级按钮字号，不因场景变化随意缩放
- 支持三种内容形式：
  - 仅文字
  - 图标 + 文字
  - 仅图标
- 图标默认出现在文字前
- 图标尺寸默认比按钮文字大 `2px`

三个层级的视觉定义如下：

- `Primary`
  - 黑色底
  - 白色字
  - 无额外描边
  - 用于当前主操作、当前激活态、最需要被看见的动作

- `Secondary`
  - 白色底
  - 黑色字
  - 灰色描边
  - 用于普通明确操作，例如确认、切换、次级面板操作

- `Tertiary`
  - 白色底
  - 黑色字
  - 无描边
  - 用于弱操作、工具条里的非当前态、低强调辅助动作

### 4.3 各层级的交互状态

所有按钮都必须具备明确的 `default / hover / active / disabled / focus-visible` 状态，不允许只定义静态样式。

状态规则如下：

- `Primary`
  - `hover`：黑底略提亮或降低压迫感，但仍保持一级按钮识别度
  - `active`：保留主按钮身份，不反转成白底
  - `disabled`：降透明度，同时取消强调感

- `Secondary`
  - `hover`：背景变浅灰，描边更清楚
  - `active`：背景比 hover 更实一些，但不升级成黑底
  - `disabled`：保留边界，降低对比度

- `Tertiary`
  - `hover`：给出轻微浅灰底，确保在白底上可识别
  - `active`：仅加强背景反馈，不新增描边
  - `disabled`：弱化文字和图标，不制造视觉噪音

### 4.4 色板按钮单独成类，不混入主按钮层级

色板按钮不属于 `Primary / Secondary / Tertiary`，必须单独维护为 `Swatch`。

原因：

- 它们承担的是“颜色选择”而不是“页面动作”
- 它们的主要识别信息来自填充色，而不是按钮底色
- 选中态更适合用描边、内阴影、对勾或聚焦环表达

色板按钮的规则：

- 默认优先显示颜色本身
- 不套主按钮的黑白底体系
- 选中态通过边框、内描边或聚焦环表达
- hover 只做轻量抬升或边框增强，不改变颜色语义

### 4.5 页面使用规则

按钮层级的使用规则固定如下：

- 一个区域里，最多只应有一个一级视觉主按钮
- 同一行里如果出现多个动作，默认从 `Secondary` 和 `Tertiary` 开始分配
- “当前选中工具”可以用 `Primary`，未选中工具默认用 `Tertiary`
- 普通确认/执行类按钮优先用 `Secondary`
- 删除、关闭、取消这类弱动作，不应默认抢成一级按钮
- 如果按钮已经只剩图标，必须保证点击面积与可访问命名完整

### 4.6 实现约束

按钮规范必须沉淀到统一组件层，而不是留在页面里口头约定。

后续实现要求：

- 主按钮统一走一个 `Button` 组件
- 色板统一走一个 `SwatchButton` 组件
- 业务组件只传 `variant`、`icon`、`pressed`、`disabled` 等状态
- 不在业务文件里直接写按钮底色、边框和 hover 规则
- 如果某个业务按钮需要特殊尺寸，应在按钮组件层做扩展，而不是局部覆盖

### 5. 能删的解释性内容就删

如果一个区域只是解释功能、但不提供新的控制能力，就优先删除。

除非信息对用户当前动作有明确价值，否则不要额外添加说明、提示词或浮层文案。

特别是这些情况应默认不加：
- 用户已经能从光标、预览、高亮、控件命名直接理解结果
- 文案会遮挡画布、工具或主要操作区域
- 文案只是重复当前工具名称、尺寸或显而易见的状态

只有在这些情况才应保留说明：
- 会影响提交结果
- 会避免明显误操作
- 会解释用户当前看不见但马上会生效的状态变化

UI 应通过这些东西帮助理解：
- 结构
- 命名
- 状态反馈

而不是依赖长文案说明。

### 6. 一次只推进一块可验证的能力

每一轮只做一块能够独立跑通的能力，例如：
- 画布缩放 / 平移 / 网格显隐
- 帧播放预览
- 图层删除 / 合并
- 拼豆色板映射
- 钩织行列 / 符号视图

不要把多个未验证能力一起推到页面上。

### 7. 先做真实可用的基础版，再做高级版

先保证主路径成立，再做增强功能。

例如：
- 先做图层显隐，再做透明度和混合模式
- 先做帧缩略图和切换，再做洋葱皮
- 先做空白画布绘制，再做选区、镜像、批量工具

### 8. 测试围绕用户动作，不围绕实现细节

优先验证：
- 点击后是否真的创建空白画布
- 点击像素后是否真的落笔
- 新建图层后是否真的可操作
- 复制帧后是否真的出现新帧

测试要证明用户功能成立，而不是只证明内部结构存在。

## 当前阶段的执行重点

后续优先顺序：

1. 钩织专属行列与符号视图
2. 拼豆与钩织的打印导出

## 当前已稳定完成的基础能力

- 空白画布创建
- 图片转绘进入文档
- 单击绘制
- 拖动连续绘制
- 填充桶
- 拖动连续擦除
- 线条工具
- 矩形工具
- 帧播放预览
- 图层基础操作
- 图层删除
- 合并到下层
- 拼豆品牌色映射
- 拼豆材料统计
- 帧缩略图时间线
- 画布缩放
- 网格显隐
- 基础平移

## 结论

当前产品方向应始终保持为：
- 先建立正确的编辑模型
- 再补真实可用的能力
- 不做只有表面形态、没有实际行为的功能
