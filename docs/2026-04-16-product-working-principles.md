# Pixel Forge Working Principles

## 核心原则

### 1. 先确认产品模型，再扩展功能

新增功能前，先判断它属于哪一层：
- 文档
- 帧
- 图层
- 画布交互
- 导出

模型不清楚时，不要先堆按钮和面板。

### 2. 功能必须闭环，不能只有入口

每个功能都必须同时具备：
- 明确入口
- 状态变化
- 用户可见结果
- 测试或命令验证

禁止只做一个按钮、一个区域或一段说明文案，而没有真实生效的行为。

### 3. 页面优先级服从工作流

界面组织顺序应始终围绕主流程：
- 先画
- 再调
- 再管理图层 / 帧
- 最后导出

说明型内容必须让位于：
- 画布
- 工具
- 图层
- 时间线
- 属性检查

### 4. 画布必须是视觉中心

后续页面调整都应遵守：
- 中间最大区域给画布
- 高频操作靠近画布
- 图层和时间线持续可见
- 不让标题、介绍、营销式文案占主视觉

### 5. 能删的解释性内容就删

如果一个区域只是解释功能、但不提供新的控制能力，就优先删除。

除非信息对用户当前动作有明确价值，否则不要额外添加说明、提示词或浮层文案。

特别是这些情况应默认不加：
- 用户已经能从光标、预览、高亮、控件命名直接理解结果
- 文案会遮挡画布、工具或主要操作区域
- 文案只是重复当前工具名称、尺寸或显而易见的状态

只有在这些情况才应保留说明：
- 会影响提交结果
- 会避免明显误操作
- 会解释用户当前看不见但马上会生效的状态变化

UI 应通过这些东西帮助理解：
- 结构
- 命名
- 状态反馈

而不是依赖长文案说明。

### 6. 一次只推进一块可验证的能力

每一轮只做一块能够独立跑通的能力，例如：
- 画布缩放 / 平移 / 网格显隐
- 帧播放预览
- 图层删除 / 合并
- 拼豆色板映射
- 钩织行列 / 符号视图

不要把多个未验证能力一起推到页面上。

### 7. 先做真实可用的基础版，再做高级版

先保证主路径成立，再做增强功能。

例如：
- 先做图层显隐，再做透明度和混合模式
- 先做帧缩略图和切换，再做洋葱皮
- 先做空白画布绘制，再做选区、镜像、批量工具

### 8. 测试围绕用户动作，不围绕实现细节

优先验证：
- 点击后是否真的创建空白画布
- 点击像素后是否真的落笔
- 新建图层后是否真的可操作
- 复制帧后是否真的出现新帧

测试要证明用户功能成立，而不是只证明内部结构存在。

## 当前阶段的执行重点

后续优先顺序：

1. 钩织专属行列与符号视图
2. 拼豆与钩织的打印导出

## 当前已稳定完成的基础能力

- 空白画布创建
- 图片转绘进入文档
- 单击绘制
- 拖动连续绘制
- 填充桶
- 拖动连续擦除
- 线条工具
- 矩形工具
- 帧播放预览
- 图层基础操作
- 图层删除
- 合并到下层
- 拼豆品牌色映射
- 拼豆材料统计
- 帧缩略图时间线
- 画布缩放
- 网格显隐
- 基础平移

## 结论

当前产品方向应始终保持为：
- 先建立正确的编辑模型
- 再补真实可用的能力
- 不做只有表面形态、没有实际行为的功能
